# 프로젝트 기능 정의 원칙

## 핵심 개념 (수정 금지)

이 프로젝트는 **IRB·DRB 승인 기반 임상연구에서 병록번호를 기준으로 수령한 R-ID와
수집 데이터 간의 정합성을 자동으로 검증·관리하는 시스템**입니다.

## 기능 정의 원칙

### 1. Master DB 생성
- 다중 수집일지 파일 병합 시 **식별번호 우선순위는 O > X > 기타** 를 항상 유지
- 같은 병록번호 내 O식별번호가 있으면 X식별번호는 반드시 가족데이터로 분류
- O식별번호가 없고 X만 있을 때는 **최신 X를 메인으로, 나머지 X는 가족데이터**로
- 병록번호 7자리는 항상 앞에 0을 붙여 8자리로 정규화
- bCODE는 **가장 최신 접수일자 기준**으로 유지, 이전 값은 HISTORY에 기록

### 2. 내부 식별키(생성ID) 생성 규칙
- 구성: `성별코드 + 출생연도(4자리) + 출생월(2자리) + 이름첫글자 + 접수일자(YYYYMMDD)`
- 성별코드: M 또는 F
- 출생연도: 2자리 입력 시 24 이하 → 2000년대, 25 이상 → 1900년대
- 이 규칙은 R-ID 매칭 로직과 반드시 동일해야 함

### 3. R-ID 매칭
- 매칭 대상: 내부 식별키가 있고 R-ID가 비어있는 행만
- 이미 R-ID가 있는 행은 재매칭하지 않고 보존
- R-ID 파일 내 동일 식별키 중복 시 해당 건은 매칭 제외 후 중복 시트로 분류
- 매칭 결과는 반드시 **매칭성공 / 미매칭 / 중복** 세 가지로 분류

### 4. 오기입 보정 매칭
- 외부 데이터소스(Hubis) 기준으로 성별·생년월 보정 후 식별키 재생성
- 보정 건수, 재매칭 성공/실패 건수를 완료 메시지에 반드시 표시
- 미매칭 잔여 건은 병록번호+접수일자 추출 파일로 자동 생성

### 5. 출력 파일 원칙
- R-ID 컬럼은 미매칭·중복 시트에서 제거
- 병록번호 컬럼은 항상 텍스트 서식 적용
- 모든 결과 파일은 입력 파일과 같은 폴더에 저장

## 코드 작성 원칙
- 함수명, 변수명, 주석은 한국어 또는 영문 혼용 허용
- 컬럼 상수(COL_*)는 파일 상단에 정의하고 하드코딩 금지
- GUI 블로킹 방지를 위해 무거운 처리는 별도 스레드에서 실행
- 진행 상태창(ProgressWindow)은 모든 기능에서 사용
- 파일 선택은 반드시 tkinter 다이얼로그 사용
